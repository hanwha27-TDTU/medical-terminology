## 기록 ① — KBG Medical Note Platform 구조 통합 착상 (event_type: design_decision)

- **기능/주제명:** KBG Medical Note Platform — 8룸 정보구조 통합
- **문제 (왜 필요한가):** 개인 의학 학습앱(Medical Note)에 어학/CPX·OSCE 학습을 통합하려는데, 기존 단일 HTML 앱의 데이터·동기화·연구노트·해시체인을 깨지 않고 확장할 구조가 없었다.
- **기존 한계:** 자매 어학앱을 통째 병합하면 Supabase·백업·연구노트·TSA가 이중화되어 충돌·오염 위험. 프레임워크 전환은 단일 HTML 배포·오프라인 장점을 잃음.
- **사람(발명자)의 아이디어·착상:** 단일 HTML을 유지하되 내부를 모듈형으로 정리하고, 최상위 8룸 정보구조(Dashboard/Medical Core/Clinical Language/CPX·OSCE/Review Engine/Unified Search/Data Management/Evidence)를 기준으로 삼는다. 신규 코드는 `KBG_MedicalNote` 단일 namespace로 소유·충돌방지. 어학은 시스템 코어를 흡수하지 말고 기능만 이식.
- **사람이 AI 결과를 수정·판단한 점:** AI의 "전체 병합" 접근을 거부하고, 어학의 별도 시스템(Supabase/백업/연구노트/TSA/테마/AI/device_id)은 이식 대상에서 제외하도록 범위를 좁힘.
- **최종 결정 (사람):** additive 방식으로만 진행. 기존 기능/DOM/저장/동기화/해시/TSA 무변경. 문서(구조 고정)를 코드보다 먼저.
- **AI 출력 요약:** Claude가 구조 분석·매핑표·위험도 평가·migration plan·namespace 도입안을 문서로 작성 보조.
- **기술적 효과:** 확장 시에도 과거 데이터·백업·해시체인 증거력 보존. 신규 코드가 전역 충돌 없이 격리됨.
- **⚖️ 기술적 의사결정 (검토→선택→이유):**
  - 검토한 기술: (A) 프레임워크 전환 (B) 어학앱 전체 병합 (C) 단일HTML+additive namespace 셸
  - 선택한 기술: C
  - 선택하지 않은 기술: A, B
  - 이유: A는 오프라인·단일파일 장점 상실, B는 시스템 이중화·연구노트 오염 위험. C는 기존 무결성 보존하며 확장 가능.
  - 장점: 무변경 additive, CI 정확명 심볼 보호와 양립, 롤백 용이.
  - 단점: 좌측 사이드바를 영구 기본 내비로 승격하려면 후속 반응형 작업 필요.
- **🎯 특허 포인트 우선순위:**
  - `1 | 단일 HTML 내부 additive namespace 셸로 기존 무결성(해시체인·서명·TSA) 보존하며 정보구조를 재구성 | 기존 방식은 재작성/프레임워크 전환으로 증거 연속성 단절`
  - `2 | canonical entity + 태그로 다중 메뉴 노출(중복 저장 없이) | 카테고리별 중복 저장 대비 데이터 일관성`
- **★ 발명자 점수:** `착상 | 5 | 0` · `구조설계 | 5 | 1` · `방향결정 | 5 | 0` · `구현 | 2 | 5` · `문서화 | 3 | 5`

## 기록 ③ — 특허연구노트 v2.0 보강 (event_type: code_change, v2.35)

- **기능/주제명:** 특허연구노트 v2.0 — 구조·알고리즘·청구항·발명자점수 + 자동 품질검사

- **기존 한계:** 해시 코어를 건드리면 과거 엔트리 재베이스라인·증거력 훼손. 필드를 무조건 추가하면 기존 기록 해시가 바뀔 위험.
- **사람의 착상:** 기존을 Base로 분석 → 부족분만 additive. 절대 수정/삭제/덮어쓰기 금지. 변리사가 거의 수정 없이 명세서 초안으로 쓸 수준 목표.
- **사람이 AI 결과를 수정·판단한 점:** AI에게 해시 코어 불변 + 조건부 스프레드 방식을 지시. 읽기 전용 분석/품질검사만 허용(기존 기록 무수정).
- **최종 결정 (사람):** 새 필드는 값이 있을 때만 저장(조건부 스프레드) → 미사용 기록 해시 바이트 동일. `_rnComputeHash`/`stableForHash`/제외목록 불변.
- **AI 출력 요약:** Claude가 `_rnBlank` 조건부 필드, 폼 그룹, `_rnMd` 섹션, `rnQualityCheck`/`rnAnalyzeNotebook`(읽기전용) 구현.
- **⚖️ 기술적 의사결정:** 검토=(무조건 필드추가 vs 조건부 스프레드) / 선택=조건부 스프레드 / 이유=미사용 기록 해시 페이로드 바이트 동일 보장(불변조건 19d 준수) / 단점=폼 입력이 텍스트 파싱 기반.
- **🏗️ Architecture:** 필드는 `data jsonb` 안에만 추가(SQL 변경 0). 공개 매니페스트는 화이트리스트라 새 내용 필드 유출 없음.
- **🔢 Algorithm:** `_rnObjHasContent(o)` → 값 있으면만 키 저장 → `_rnComputeHash`는 기존 로직 그대로 → 미사용 시 이전과 동일 직렬화.
- **기술적 효과:** 특허 요소 커버리지·10항 품질검사 자동화. 과거 기록·서명·TSA 증거력 영향 0(브라우저로 `emptyHasNoNewKeys=true` 검증).
- **테스트 방법/결과:** golden `_rnComputeHash` 고정해시 불변 통과 + 브라우저: 미사용 해시 바이트동일·품질 10/10·소급 필수문구·분석 모달·페이지에러 0.
- **🎯 특허 포인트 우선순위:** `1 | append-only 특허 증거 로그에 조건부 스프레드로 필드를 additive 확장해 과거 해시 불변을 보장하는 방법 | 일반 스키마 확장은 기존 해시/서명 무효화`
- **📜 청구항 후보:** `해시 체인으로 봉인된 기록 로그에서, 신규 선택적 필드를 값 존재 시에만 직렬화 페이로드에 포함시켜 기존 기록의 해시 불변성을 유지하며 스키마를 확장하는 방법`
