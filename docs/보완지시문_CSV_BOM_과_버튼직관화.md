# 어학 마스터앱 보완 지시문 — ① CSV 한글 깨짐(UTF-8 BOM) ② 정정 버튼 직관화

> 자매앱(의료 노트)에서 방금 적용한 두 가지 개선을 어학 마스터앱에도 이식하기 위한 지시문.
> 함수명은 참고용(그쪽 앱 실제 이름에 맞게 대응). **"왜"까지 이해하고 적용할 것** — 맹목 복붙 금지.
> 각 항목은 독립 커밋/PR, 버전 +0.01, 업데이트 이력·스킬 문서에 교훈 기록.

---

## ① CSV를 Excel에서 열면 한글이 깨진다 → UTF-8 BOM 부착

### 증상
- 앱에서 내보낸 CSV를 **윈도우 한글 Excel**로 열면 한글이 `源遺媛`처럼 깨져 보인다.
- 단, **JSON·Markdown**으로 내보내면 멀쩡하고, 텍스트 에디터(VS Code 등)로 CSV를 열어도 정상이다.

### 원인 (오해 금지)
- **파일이 깨진 게 아니다.** CSV 내용은 UTF-8로 정상 저장돼 있다.
- 윈도우 Excel이 CSV를 열 때 **인코딩을 UTF-8로 자동 인식하지 못하고** 시스템 기본 코드페이지(한국어=CP949/EUC-KR)로 읽어버려서 깨진다.
- 즉 **"데이터 문제"가 아니라 "Excel의 인코딩 자동감지 실패"** 문제다.

### 해결 — 파일 맨 앞에 UTF-8 BOM(`﻿`)을 붙인다
BOM(Byte Order Mark)이 앞에 있으면 Excel이 "이건 UTF-8"이라고 즉시 인식한다.

**모든 CSV 다운로드 경로**에서, 텍스트 맨 앞에 `﻿`를 붙인다:

```js
// 다운로드 헬퍼에서 CSV면 BOM 부착(이미 있으면 중복 금지). JSON/MD 등엔 붙이지 않는다.
function downloadFile(name, text, mime) {
  let out = String(text);
  if ((/csv/i.test(mime || '') || /\.csv$/i.test(name)) && out.charCodeAt(0) !== 0xFEFF) {
    out = '﻿' + out;               // ← UTF-8 BOM
  }
  const blob = new Blob([out], { type: mime || 'text/plain;charset=utf-8' });
  const a = document.createElement('a'); a.href = URL.createObjectURL(blob); a.download = name;
  document.body.appendChild(a); a.click(); setTimeout(() => { URL.revokeObjectURL(a.href); a.remove(); }, 500);
}
```

- **핵심:** CSV·TSV(Excel로 여는 표 형식)에는 BOM을 붙이고, **JSON·Markdown·순수 텍스트에는 붙이지 않는다**(그쪽에선 BOM이 오히려 첫 글자로 섞여 문제를 만들 수 있다).
- 헬퍼가 여러 개면 **모든 CSV 내보내기 경로**를 점검한다. (자매앱 의료 노트에선 대부분의 CSV는 이미 BOM을 붙이고 있었는데 연구노트 다운로드 한 곳만 누락돼 있었다 → **한 곳이라도 빠지면 그 파일만 깨진다.**)
- 소스에는 **눈에 안 보이는 리터럴 BOM 문자 대신 `'﻿'` 이스케이프**를 쓴다(가독성·툴이 실수로 지우는 것 방지).

### 검증
1. 한글이 든 데이터를 CSV로 내보내 **윈도우 Excel에서 열어** 한글이 정상인지 확인(가장 확실).
2. 코드 레벨: CSV 결과 문자열의 `charCodeAt(0) === 0xFEFF`인지, 이미 BOM이면 중복 안 붙는지, JSON엔 안 붙는지 단위 확인.
3. `node --check`(단일 HTML이면 `<script>` 블록 추출 후).

### (사용자 안내용) 이미 받아둔 옛 파일을 다시 볼 때
새로 내려받는 게 제일 간단. 굳이 옛 파일이면 Excel `데이터 → 텍스트/CSV 가져오기 → 원본 "65001: 유니코드(UTF-8)"` 지정.

## ② append-only / 정정 버튼이 "누르면 뭐가 되는지" 안 보인다 → 직관화

### 증상 / 문제
- 연구노트(또는 정정 가능한 기록) 카드의 버튼들 — 예: `상세` `특허 후보` `사람 결정` `정정` `삭제표시` — 이 **상태 태그처럼** 보여, 사용자가 "이걸 누르면 무슨 일이 일어나지?"를 라벨만으로 알 수 없다.
- 특히 원본 불변(append-only) 설계에선 이 버튼들이 **원본을 고치는 게 아니라 "원본을 보존한 채 연결된 새 기록을 추가"**하는데, 그 의미가 안 드러나면 **파괴적 동작으로 오해**할 수 있다.

### 해결 — "행동형 라벨 + 공유 원칙 한 줄 + 버튼별 툴팁" 3층

1. **라벨을 행동형으로** (명사/태그 → 무엇을 하는지):
   | 이전(태그처럼) | 이후(행동형) |
   |---|---|
   | 상세 | 🔎 상세보기 |
   | 특허 후보 | 🏷️ 특허감으로 표시 |
   | 사람 결정 | ＋ 내 결정 덧붙이기 |
   | 정정 | ↻ 정정(수정본 추가) |
   | 삭제표시 | 🗑️ 삭제표시 |
   (어학앱 맥락에 맞게 문구 대응 — 핵심은 "동사/결과"가 보이게.)

2. **버튼 그룹 위에 공유 원칙 한 줄**:
   ```
   🔒 원본은 못 고칩니다 — 아래 버튼은 원본을 보존한 채 연결된 새 기록을 추가합니다 (상세는 열람만).
   ```
   → 5개 버튼의 공통 규칙을 한 번에 이해시킨다.

3. **버튼마다 `title` 툴팁**(마우스 올리면 뜨는 한 줄 설명):
   ```html
   <button title="이 기록의 전체 내용·해시·서명·타임스탬프를 봅니다 (열람만, 변경 없음)">🔎 상세보기</button>
   <button title="원본은 그대로 두고, 이 주제에 대한 새 기록을 추가합니다 (원본 보존)">＋ …</button>
   <button title="내용을 고칠 때 — 원본은 못 고치므로 수정본을 새로 추가합니다 (원본·해시 보존)">↻ 정정…</button>
   <button title="원본·해시는 보존되고, 삭제 사실도 기록으로 남습니다 (조용한 삭제 없음)">🗑️ 삭제표시</button>
   ```

4. **클릭 후 폼 상단 안내를 유형별로 구체화**: 정정 폼을 열면 "이것은 **○○ 기록**입니다. 원본은 그대로 보존되고, 이 항목이 원본에 연결된 새 기록으로 추가됩니다." 를 `correction_type`(amend/decision/patent/delete 등)에 맞춰 문구를 바꿔 보여준다.

### 원칙 (일반화 교훈)
> **append-only·정정 UI는 "무엇이 되는가"가 라벨만으로 안 보이면 사용자가 파괴적 동작으로 오해한다.**
> → 라벨은 **행동형**, 공유 원칙은 **버튼 근처 한 줄**, 세부 동작은 **툴팁**. 이 3층이면 처음 보는 사람도 실수 없이 이해한다.

### 검증
- 각 버튼 툴팁이 뜨는지, 라벨이 행동을 나타내는지.
- 버튼 위 안내 한 줄이 그룹 전체 규칙을 설명하는지.
- `node --check`(단일 HTML이면 `<script>` 블록).
- **삭제/정정이 원본을 실제로 안 건드리는지**(append-only 불변) 회귀 확인.

## 공통 마무리 체크리스트
- [ ] CSV 내보내기 **모든 경로**에 BOM(단, JSON/MD엔 금지). 소스는 `'﻿'` 이스케이프.
- [ ] 정정 계열 버튼: 행동형 라벨 + 공유 원칙 한 줄 + 툴팁 + 유형별 폼 안내.
- [ ] 각 사안 버전 +0.01, 업데이트 이력 항목, 스킬/문서에 **"왜/함정"**까지 기록(버전 번호만 남기지 말 것).
- [ ] `node --check` 통과. (BOM은 Excel 실열어보기, 버튼은 툴팁·원본 불변 확인.)
