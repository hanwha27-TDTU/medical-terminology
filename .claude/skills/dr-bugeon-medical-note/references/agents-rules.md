# Supabase Sync Instructions

## 디자인·연결 회귀 방지

- 빠른 참조용 모달, 유용한 사이트, 외부 검색, 데이터/가이드/퀴즈 런처처럼 "잠깐 열어 확인하는" 화면은 데스크톱에서 내부 스크롤 없이 핵심 항목이 한 화면에 들어오도록 먼저 설계한다. 카드 높이, 제목 크기, 설명 줄 수, 버튼 크기를 줄여도 사용 목적이 빠른 선택이면 조밀한 그리드가 우선이다.
- 라이트모드 디자인은 단순히 다크모드 색을 반전하지 않는다. 큰 원색 텍스트, 굵은 테두리, 과한 그림자, 컬러 이모지가 동시에 보이면 조잡해 보이므로 낮은 채도, 얇은 테두리, 작은 액션 버튼, 충분한 여백으로 다시 조정한다.
- 새 카드/상세/모달 UI를 만들 때는 label/value를 큰 글씨로 단순 나열하지 않는다. 학습 앱에서는 상단 핵심 요약, 암기 포인트, 비교 포인트, 세부 정보, 출처/외부 링크 순으로 시선 흐름을 정리하고 줄간격과 글자 굵기를 단계화한다.
- 팝업이나 외부 링크를 "별도 창"으로 열어야 하는 기능은 단순 `<a target="_blank">`만 믿지 않는다. `window.open(url, name, features)`를 우선 사용하고, 팝업 차단이나 실패 시 새 탭 fallback을 제공한다. 검증 시에는 `window.open`을 stub으로 잡아 URL, 창 이름, features가 의도대로 전달되는지 확인한다.
- 앱 내부 자동 링크는 검색창으로 우회하지 말고 가능한 경우 해당 도메인의 상세 드로어/상세 모달을 직접 연다. 단, 연결 대상이 없거나 중복 후보가 있으면 검색 결과나 후보 선택 UI로 부드럽게 fallback한다.
- 새 데이터 도메인을 추가하면 목록 표시만으로 완료하지 않는다. 추가/편집, 상세 드로어, 모바일 bottom sheet, 즐겨찾기, 발음/IPA 등 공통 학습 기능, 복사 기능, TSV/Excel 가져오기, 백업/복원, 휴지통, 통계, 퀴즈, 최종본 저장, 저장본 버전 확인까지 연결 여부를 한 번에 점검한다.
- 데이터 관리와 통계 화면은 새 도메인을 가장 자주 누락하는 곳이다. 도메인을 추가하거나 이름을 바꾸면 내보내기, 가져오기, 전체 JSON, 영역별 백업, 복원, 일괄 삭제, tombstone, 동기화 안내 문구, 항목 수 카운트를 반드시 함께 갱신한다.
- 객관식/오답노트 파서는 정답이 하나라고 가정하지 않는다. `+` 정답 줄이 여러 개일 수 있으므로 저장 구조는 다중 정답 배열을 지원하고, 기존 단일 정답 필드가 있으면 backward compatibility를 둔다. UI 선택, 저장, 복원, 보기 화면이 모두 같은 정답 표현을 사용해야 한다.
- 빠른 입력 파서, TSV/Excel import, AI 생성 결과처럼 한 번에 여러 필드를 채우는 기능은 파싱 결과를 화면에 그리는 것뿐 아니라 실제 저장 객체와 복원 객체까지 확인한다. "보이는 상태"와 "저장되는 상태"가 다르면 회귀로 본다.
- 디자인 수정 후 브라우저 검증은 스크린샷만 보지 말고 수치도 확인한다. 주요 모달은 `scrollHeight <= clientHeight` 여부, `scrollWidth <= innerWidth`, 마지막 카드의 viewport 포함 여부, 닫기/열기 버튼 클릭 가능 여부를 데스크톱·태블릿·모바일 폭에서 확인한다.
- 내부 상태 검증이 필요한 단일 HTML 앱은 Playwright가 top-level `const`/`let`을 `window`에서 읽을 수 있다고 가정하지 않는다. 테스트가 필요한 상태는 의도적으로 `window.__APP_TEST_HOOKS__` 같은 read-only hook이나 DOM 신호로 노출한다.
- UI-only 변경이라도 외부 링크, 데이터 도메인, 저장 구조, import/export, Supabase 동기화 경로에 닿았는지 확인한다. 완료 보고에는 SQL 변경 필요 여부와 지침 업데이트 여부를 함께 남긴다.

## 완료 보고 (항상 마지막에)

- 모든 작업은 **마지막에 결과 보고**로 마무리한다(사용자 상시 요청). 최소 포함: ① 무엇을 바꿨나(핵심만), ② 검증 결과(JS 구문검사/렌더링 등 *실제로 한 것*과 그 결과), ③ 산출물(파일 **전체 절대경로를 코드블록**으로 — 한글·공백 경로라 마크다운 링크는 클릭 안 됨), ④ `SQL 변경 필요: 있음/없음/불확실`, ⑤ `지침 업데이트: <문서명>/없음`, ⑥ 배포했으면 배포 결과·Pages URL(또는 사용자가 직접 push해야 하는지).
- ⑤ **`지침 업데이트`는 빈칸으로 두지 않는다 — 사용자 지시가 없어도 매 작업 자동·100% 점검(누락 금지).** 새 교훈·규약·실수·패턴·설계결정이 하나라도 있으면 해당 스킬 문서에 반영하고 어느 파일에 무엇을 넣었는지 적는다. 정말 없으면 `지침 업데이트: 없음(새 교훈 없음)`이라고 명시해 "점검했으나 없음"과 "빠뜨림"을 구분한다. (SKILL.md 핵심 원칙 #9 — 스킬 갱신은 definition-of-done.)
- 못한 것·건너뛴 것·실패한 검증은 숨기지 말고 그대로 적는다. "했다"는 검증으로 확인된 것만.

## 입력 보조 UI 회귀 방지

- 특수문자 입력기, 수식 입력기, 템플릿 삽입기처럼 현재 입력칸에 텍스트를 넣는 공용 도구는 `textarea`와 명확한 텍스트형 `input`에만 적용한다. `type="number"` 같은 브라우저 검증 입력칸은 비숫자 삽입 거부와 `setSelectionRange` 예외가 생길 수 있으므로 대상에서 제외한다.
- 입력 보조 UI는 현재 포커스된 입력칸이 없을 때 마지막 입력 대상을 재사용할 수 있지만, 그 노드가 `document.contains()`로 현재 문서에 붙어 있는지 확인한다. 닫힌 모달이나 다시 렌더링된 입력칸처럼 detached node에는 삽입하지 않는다.
- 커서 복원용 `setSelectionRange`는 입력 타입과 브라우저에 따라 예외가 날 수 있으므로 지원 여부를 확인하고 필요하면 `try/catch`로 감싼다. 삽입 후에는 `input` 이벤트를 발생시켜 미리보기, 자동완성, 저장 버튼 상태가 함께 갱신되게 한다.
- 용어, 약물, 공식처럼 같은 앱 안의 병렬 입력 도메인에 공통으로 필요한 입력 도구는 도메인마다 따로 만들지 말고 하나의 공용 헬퍼로 만들되, 모달 안에서 중복 노출되지 않도록 위치를 조정한다.
- 모달, drawer, bottom sheet처럼 자체 스크롤·transform·backdrop·z-index가 있는 컨테이너 안에서 데스크톱 플로팅 패널을 열 때는 컨테이너 내부 `position: fixed`에 의존하지 않는다. viewport 기준 위치가 필요하면 body 직속 포털 패널로 렌더링하고, 열림/닫힘/resize/scroll/focus 이동 때 위치와 표시 상태를 함께 관리한다.
- 같은 입력기 UI가 inline 패널과 body 포털 패널을 동시에 가질 때는 테스트와 삽입 로직이 숨겨진 템플릿 패널을 잡지 않게 한다. `display`, `visibility`, 실제 rect를 기준으로 현재 보이는 활성 패널만 검증하고 조작한다.

## 최근 회귀 방지 규칙

- **단일 HTML 앱을 구조적으로 확장할 땐 기존 코드를 옮기지 말고 additive namespace(신규 코드 전용)로 얹는다.** 정보구조 재편·새 상위 내비게이션 등은 새 최상위 namespace(예: `KBG_MedicalNote.AppRouter/AppUI`)에 두고 **기존 함수를 "호출만"** 한다. 기존 함수/DOM id/상수를 namespace 안으로 이동·rename하면, 정확명 심볼을 추출하는 CI(check-index·golden)가 즉시 깨지고 inline `onclick` 핸들러도 끊긴다. 뷰/라우팅 계층 추가는 저장소·데이터를 건드리지 않는다(방=뷰, 저장은 단일 canonical entity).
- **단일 HTML "디자인 전면 개편"은 마크업을 옮기지 말고 CSS로 감싸라(reframe).** 개념도형 좌측 사이드바+상단바는 기존 `<header>`/본문을 그대로 두고, 고정 사이드바/상단바를 얹은 뒤 `body{padding-left/padding-top}`로 기존 앱을 밀고, 셸에서만 기존 sticky 헤더를 `position:static`으로 흐르게 한다. 비-라이브러리 룸은 메인 영역만 덮는 패널로. **미디어쿼리 폴백**: 셸은 태블릿·데스크톱·**폴더블 펼침**(≥600px, v2.61에 768→600 하향 — 폴더블 펼침 670~900px가 오버레이 대신 프레임을 받도록), 폰 세로(<600px)는 기존 화면+오버레이(사이드바/상단바 숨김·padding 0). 분기점은 **딱 2개 미디어쿼리(min-width/max-width)**에만 있으니 바꿀 땐 둘 다 + 아래 hscroll 클립까지 같은 값으로. **⚠️ 두 함정(둘 다 실제 발생):** ① `body` padding 때문에 **화면 밖 fixed 드로어**가 `html`(overflow-x:visible)에서 문서를 가로로 늘려 **셸이 켜지는 좁은 폭에서만** 가로스크롤 → `@media(min-width:셸폭){html{overflow-x:hidden}}`. ② 슬라이드 패널/드로어를 `openDrawer()` **직후 동기 측정하면 애니메이션 전(화면 밖) 위치가 잡혀 오탐** → `waitForTimeout(~500ms)`(또는 `transitionend`) 후 측정.
- **여러 PR을 촘촘히 연속 머지하지 말 것 — 레거시 GitHub Pages 배포가 동시성으로 실패한다.** 이 저장소는 "Deploy from a branch"(워크플로 `on: dynamic`, concurrency group 없음)라, main에 몇 분 내 연속 머지하면 뒤 배포가 `##[error]Deployment failed, try again later`로 실패한다(build는 성공, deploy만 빨강 — 코드 문제 아님). **예방:** 변경을 한 PR로 묶거나 앞 배포 완료 후 다음 머지. **복구:** 실패한(최신) run의 `rerun_failed_jobs` 재실행이면 즉시 성공(코드 수정 불필요). 상세: install-deploy-guide §7.7.
- **모바일 1차 내비게이션은 가로 스크롤 띠로 두지 말 것(대상이 화면 밖에 숨는다).** 방/탭이 많으면(예: 8룸) `flex-direction:row;overflow-x:auto`는 뒤쪽 항목이 잘려 발견성이 나쁘다 → **줄바꿈 그리드**(`display:grid;grid-template-columns:repeat(auto-fit,minmax(~118px,1fr))`)로 한 화면에 다 보이게. 전체화면 오버레이는 세로 공간이 넉넉하니 몇 줄 차지해도 스크롤 숨김보다 낫다. (v2.62 플랫폼 오버레이)
- **화면폭 검증은 양극단(폰·데스크톱)만 보지 말고 "중간 밴드(태블릿·좁은 데스크톱 800~1200px)"를 반드시 포함하라.** 실제로 이 앱은 데스크톱·모바일만 검증했다가 놓친 가로스크롤 버그가 **1101~1200px 구간에서만** 났다(셸 켜지는 최소폭 근처). Playwright 매트릭스는 최소 `[390, 768, 1024, 1120, 1440] × [dark, light]`로, 각 폭에서 `document.scrollingElement.scrollWidth <= innerWidth+1`(가로스크롤 없음)와 셸 on/off 전환폭을 확인한다.
- **로고/이미지를 단일 HTML에 넣을 땐 원본 그대로 base64하지 말고 먼저 리사이즈한다.** 이 환경엔 PIL/sharp/convert가 없으니 **Chromium(Playwright) canvas로 리사이즈** → `toDataURL`. 사진/그라데이션 포함 로고는 **PNG 대신 JPEG**(흰 배경 채우고 q~0.82)로 — 실제 1819×865 로고가 460px PNG 172KB → JPEG 15KB로 줄었다. 큰 base64는 Edit 대신 Node 스크립트로 상수 주입. **브랜드 표기 변경은 표시 전용** — `APP_INFO.name`·`RN_PROJECT_NAME`은 절대 불변(연구노트 격리·백업 식별; 바꾸면 과거 기록 숨김·호환 파손).
- **노트 등 "연결(link) 필드"를 늘리면 전 지점을 함께 갱신하고, 링크는 충돌판정에서 제외 + 병합 union으로 보존한다.** 링크 배열은 자동 파생이라 충돌 트리거로 쓰면 스퓨리어스 "충돌 사본"이 양산된다 → `noteContentDiffers`에서는 제외, `merge*ForSync`에서 합집합 보존, 해시 페이로드엔 포함(스냅샷 전파). 새 링크 종류 추가 시 normalize 화이트리스트·검색·필터·뷰·편집폼·저장 autoLink·통계·CSV·해시 payload·union 병합 + 백업(수동 검증, jsonb는 자동 게이트 없음)까지 한 번에. (reconstruction-spec 불변조건 10·§6.1)
- **폐기되는 자매 앱을 흡수할 땐 "새 도메인 IIFE"로 얹고 코어 함수는 몽키패칭 체이닝으로 확장한다(additive).** 별도 앱(예: 어학앱) 데이터를 단일 HTML로 흡수할 때, 기존 노트(jsonb-blob) 도메인 패턴을 미러링한 **자기완결 IIFE**(저장키·`window.<DOMAIN>` 배열·`normalize*`·load/save·tombstone)를 새로 추가하고, 백업/복원/미리보기 코어(`createCompleteBackupObject`/`readDataImportFile`/`applyPendingDataImport`/`calculateImportPreview`/`importPreviewCountsHtml`)는 **원본을 캡처해 다시 감싸는(re-wrap) 체이닝**으로 확장한다(노트 IIFE가 이미 쓰는 패턴 → 순서상 뒤 IIFE가 앞 패치를 감싸 자동 체이닝). ⚠️ `applyPendingDataImport`는 반환 전 `pendingDataImport=null`로 만들므로 **원본 호출 전에** 필요한 페이로드를 캡처한다. 신규 컬럼-스키마 게이트(check-schema-drift/restore-drift/index parity)는 jsonb-blob 도메인을 대상에서 제외하므로, SQL(Supabase) 없이 로컬 저장 계층만 먼저 얹어도 게이트가 깨지지 않는다.
- **복원 데이터의 무손실 이식은 "왕복검증부터, 코드/스키마는 나중"이라는 순서로 한다.** 원본→내부모델→원본이 byte-identical인지 먼저 증명(정렬 canonical 문자열 비교)한 뒤에 UI·스키마를 얹는다. 임포터는 **UTF-8 BOM 제거**(`charCodeAt(0)===0xFEFF`면 `slice(1)`), **id는 문자열 그대로 보존**(짧은 id/UUID 혼재 정상, 재발급 금지), **learning 등 중첩 상태 verbatim**, null 필드(예: `audio_url`)는 null로 보존(''로 강제하면 왕복 깨짐). SQL은 한 번에 마지막에 통합.
- **저장소 역할을 고정한다: Supabase = 텍스트 + 오디오 녹음(Storage 버킷), Cloudinary = 이미지 + 영상.** 오디오는 이미지/영상 변환 파이프에서 얻을 게 없고, 텍스트와 같은 백엔드에 둬야 인증·수명주기·백업·삭제(tombstone)가 통일된다. 녹음은 `MediaRecorder`로 캡처(webm/opus 우선)하고 URL만 DB에 저장(바이트 DB 금지).
- **흡수한 데이터는 앱에 박아 배포하지 않는다 — 앱은 빈 상태로 두고 사용자가 앱 복원기능으로 직접 주입한다.** 소스코드에 데이터 스냅샷을 커밋하면 앱이 무거워지고 그 시점에 고정된다. 개발자는 임포터·복원 UI·표시 틀만 만들고, 방을 처음 열면 복원 CTA가 뜨게 한다. 복원 경로가 여러 개면(전용 버튼 + 표준 가져오기) 어느 쪽으로 넣어도 되게 한다.
- **인라인 `onclick` 속성에 문자열을 `JSON.stringify`로 보간하지 말 것(따옴표 파손).** `JSON.stringify(text)`는 항상 큰따옴표로 감싸므로 **큰따옴표 속성** 안에 넣으면(`onclick="fn(${JSON.stringify(text)})"`) 첫 내부 큰따옴표에서 속성이 조기 종료돼 핸들러가 통째로 깨진다(전 항목 무동작). 규칙: **텍스트는 속성에 넣지 말고 id만 넘겨 핸들러 내부에서 조회**한다(골드 레퍼런스 `speakTermField(event,id,…)` 패턴). 문자열 id를 부득이 넣을 땐 **작은따옴표 속성 + `JSON.stringify`의 `'`를 `&#39;`로 치환**한 헬퍼(`_j`)를 쓴다(JSON.stringify는 작은따옴표를 이스케이프하지 않아, 외부 데이터에서 온 임의 문자열 id에 `'`가 있으면 작은따옴표 속성이 깨진다). 숫자 id 도메인은 안전하지만, 외부 앱에서 흡수한 **문자열 id 도메인**은 이 함정에 특히 취약.
- **새 도메인의 "전체 교체(replace) 임포트"는 제거분에 tombstone을 남기고, "복원 임포트"는 그 도메인의 tombstone을 반드시 되읽어야 한다.** replace가 사라진 레코드에 tombstone을 안 남기면(또는 백업의 `<domain>Tombstones`를 export만 하고 import 경로에서 안 읽으면), 그 삭제가 다음 동기화에서 클라우드 active본으로 **부활**한다. 의학 도메인의 `removed=…filter(!incomingIds).map(recordLocalTombstone)` + 복원 시 `save*Tombstones` 미러를 새 도메인에도 그대로 복제한다. "tombstone을 쓰기(백업)만 하고 읽기(복원)가 없다"가 신규 도메인의 전형적 누락.
- **새 도메인을 추가하면 저장·동기화뿐 아니라 전역 크로스도메인 기능에도 수평전개한다.** "전체를 다루는" 기능 — 통합검색(전체/도메인탭), 대시보드 라이브러리 현황·수치 카드, 플래시카드(전 도메인 랜덤 능동회상), 기기 동기화 상태 패널 — 에 새 도메인이 빠지면 "한 도메인만 빠짐" 불일치가 사용자에게 그대로 노출된다(예: 어학 흡수 후 통합검색·대시보드·플래시카드에 어학 누락 → v2.59 수평전개). 단, 데이터 성격이 다르면 무관한 기능은 제외해도 된다(예: 어학은 엔티티 연결맵 대상이 아니라 통합검색 결과의 🔗을 `noMap`으로 생략). 새 도메인 커밋 전 "이 도메인이 나와야 할 전역 화면"을 열거해 대조한다.
- **새 도메인을 추가하면 자동 안전망(golden·check-index)도 함께 확장한다.** 단일 HTML 앱의 유일한 지속 회귀 가드는 이 게이트들인데, 새 도메인이 여기서 빠지면(감사에서 실제로 드러남) 리팩토링·삭제·리네임 회귀를 사람 감사로만 잡게 된다. 최소: ① `golden-tests.mjs`에 그 도메인 핵심 순수함수(정규화 왕복 등)의 골든 잠금, ② `check-index-scripts.mjs` REQUIRED_SYMBOLS에 도메인 코어 심볼(저장·정규화·임포트·동기화·방·미디어)을 "없으면 확실히 문제" 기준으로 추가. 순수함수에 기본파라미터 객체(`(r={})`)가 있으면 표준 `extractSource`(첫 `{`=본문 가정)가 깨지니 파라미터 리스트를 건너뛰는 추출기를 쓴다.
- **정규화(normalize) 화이트리스트는 편집 가능한 학습/상태 하위필드를 하나도 빠뜨리면 안 된다(왕복 유실).** 예: SRS `learning.reviewedAt`(마지막 복습 시각)를 normalize가 버리면, 저장·동기화·재로드 왕복마다 소실돼 방금 복습한 카드가 즉시 다시 due로 뜬다(레벨은 살아도 간격 붕괴). 단, **원본 백업에 없던 필드는 조건부로만 추가**해 무손실 왕복(byte-identity)을 깨지 않게 한다(값 있을 때만 스프레드).
- 퀴즈, 복습, 추천, 랜덤 출제처럼 순서 무작위화가 학습 품질에 영향을 주는 곳에서는 `array.sort(() => Math.random() - 0.5)`를 쓰지 않는다. 균등 셔플이 필요하면 Fisher-Yates 셔플을 사용하고, 취약/미완성/복습 예정 항목처럼 우선순위가 의미 있는 목록은 셔플하지 않는다.
- 외부 JSON, 구버전 백업, CSV/Excel 등에서 tombstone을 가져올 때 `updated_at`/`updatedAt`이 없고 `deleted_at`/`deletedAt`만 있으면 삭제 시각을 `updated_at`으로 보정한다. 가져오기 시각으로 임의 승격하면 정상 레코드를 잘못 숨길 수 있다.
- 통계, 퀴즈, 학습 현황, 필터 카운트처럼 사용자에게 현재 항목 수를 보여주거나 학습 대상을 고르는 로직은 tombstone/soft delete 항목을 제외한 활성 항목 기준으로 계산한다.

## 데이터 유실 회귀 방지 (2026-06 전수감사 일반화)

전수 로직감사에서 실제로 나온 결함을 모든 앱에 적용할 짧은 원칙으로 일반화한 것. (앱별 구체 위치/함수는 스킬 `references/reconstruction-spec.md` §10 불변조건 참조.)

- **삭제기록(tombstone)은 동기화 전처리(fence/필터)에서 버리지 않는다.** "기준본 이후 오래된 로컬 전용 항목 제외" 같은 fence는 *활성 행에만* 적용하고, `deletedAt` 행은 무조건 병합까지 통과시킨다. tombstone을 먼저 걸러내면 삭제가 다른 기기에서 부활한다.
- **변경 감지/충돌 판정의 비교 필드셋은 무결성 해시 페이로드와 항상 동일하게 맞춘다.** 한쪽(예: 충돌판정)이 비교 필드를 일부 빠뜨리면 "해시는 다른데 충돌은 같다"는 모순으로 편집이 유실된다. 링크/연결 배열이 여러 종류면 *전부* 비교한다.
- **무결성 snapshot 해시 페이로드에는 타임스탬프만 빼고 사용자 편집 가능한 내용 필드를 전부 넣는다.** 한 필드라도 빠지면 그 필드 단독 수정이 "일치"로 오판돼 전파되지 않는다.
- **삭제기록 운영(오래된 tombstone 정리·보호기록 개수·전체삭제)·기기 일치 비교·로더 정규화는 전 도메인을 빠짐없이 처리한다.** 도메인을 복붙으로 늘리면 "한 도메인만 누락"이 가장 흔한 결함이다. 추가 후 `rg`로 전 도메인 키가 모든 운영 경로에 나타나는지 대조한다.
- **외부 스토리지 이미지의 "삭제 대상" 산정 시 보호집합은 메타 배열 + 본문(HTML) 임베드 참조의 합집합이다.** 본문에 남은 참조를 빠뜨리면 표시 중인 원본이 삭제된다.
- **IndexedDB로 쓰고 localStorage를 지운 데이터를 동기 경로에서 localStorage로 직접 읽지 않는다.** 시작 시 메모리 캐시 로드 후 그 값을 반환한다.
- **가져오기 미리보기 카운트는 실제 적용 모드(append=신규만, merge=updated_at 비교)와 같은 규칙으로 계산한다.** 표시 수치와 실제 기록이 어긋나면 회귀로 본다.
- **간격반복(SRS) 간격은 직전 간격을 기반으로 키우고, 숙달 판정은 lifetime 누적이 아니라 연속 정답 streak/최근 정답률 기반으로 둔다.** lifetime 카운터 캡만으로 간격을 정하면 숙달 항목이 영원히 짧은 주기로 재출제된다.
- **이스케이프-언이스케이프는 대칭 순서로.** 직렬화에서 구분자를 escape했다면 파싱은 split *전에* unescape(또는 보호 토큰 치환)한다. split 후 unescape하면 셀/항목이 쪼개진다.
- **seed/기본 골격 자동 주입은 "신규 추가" 경로에서만.** 빈 섹션 편집 시에도 주입하면 매 편집마다 유령 행이 누적된다.
- **존재하지 않는 입력 요소를 읽지 않는다.** 입력칸을 폼에서 제거했으면 제출 로직의 해당 read도 제거한다(빈 값 강제 + fallback 무력화 방지).

Supabase 기기 동기화 앱은 단순 양방향 병합만으로 처리하지 않는다. 일반 동기화와 최종본 기준 동기화를 반드시 구분한다.

## 지침 파일 자동 구분

앱개발 중 새로 남길 규칙이 생기면 `AGENTS.md`와 앱별 스킬 파일에 넣을 내용을 자동으로 구분해 업데이트한다.

- `AGENTS.md`: 모든 앱개발에 반복 적용할 짧은 필수 규칙, 금지 규칙, 완료 보고 규칙을 둔다.
- 앱별 스킬 파일: 특정 앱의 구조, 화면, 함수, 테이블, 검증 시나리오, 배포 방식처럼 앱에 종속된 상세 절차를 둔다.
- 같은 교훈이 전역 규칙과 앱별 절차에 모두 걸리면 `AGENTS.md`에는 핵심 원칙만, 스킬 파일에는 구체 절차만 둔다.
- 일회성 작업 내용, 임시 파일명, 이미 끝난 문제의 장황한 설명은 지침 파일에 넣지 않는다.
- 완료 보고에는 `지침 업데이트: AGENTS.md / 스킬 파일 / 없음` 중 해당 사항을 명시한다.
- 업데이트된 앱 파일을 제공할 때는 파일명에 날짜_시간_앱이름_업데이트내역을 드러내고, 앱의 개발자 정보 또는 업데이트 이력에도 같은 변경 내역을 남긴다.
- 파서, 데이터 카탈로그, 자동판독 로직처럼 사용자가 실행 버전을 구분해야 하는 변경은 파일명과 개발자 정보뿐 아니라 앱 내부 `APP_VERSION`/파서 버전/캐시 진단 표시도 함께 갱신한다.
- 웹앱 개발 시 다크모드와 라이트모드를 함께 고려하고, 가능하면 CSS 토큰 기반으로 두 모드가 동시에 유지되게 설계한다.
- 개발자 정보에는 앱 이름·버전, 코드 최종 수정 시각, 현재 파일명·저장본, 업데이트 이력을 표시한다. 파일명을 직접 읽지 못하는 환경에서는 `앱이름_업데이트내역` 형태의 fallback을 보여준다.
- 업데이트 이력이 길어지면 최신 1~2개만 기본 노출하고 과거 이력은 접기/펼치기로 묶는다. 개발자 정보 모달의 글자 크기와 카드 여백은 본문 대시보드보다 조밀하게 유지한다.

## 샌드박스 밖 파일 제공

- 대상 폴더가 워크스페이스 밖이라 쓰기가 막히면 사용자가 직접 복사하게 하지 말고, 먼저 워크스페이스 안에 결과물을 만든 뒤 대상 경로를 검증하고 승인 실행으로 복사한다.
- 바깥 경로로 복사할 때는 `safe_publish_file.ps1`처럼 source가 워크스페이스 내부인지, destination이 드라이브 루트가 아닌지, destination parent가 존재하는지 확인하는 도우미를 사용한다.
- Windows에서 `.ps1` 실행이 정책에 막히면 사용자가 정책을 바꾸게 하지 말고 `safe_publish_file.cmd` 또는 `powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\safe_publish_file.ps1 ...` 형태로 실행한다.
- 승인 실행이 필요한 `Copy-Item`, 배포, 브라우저 열기 같은 작업은 실패 후 같은 명령을 반복하지 말고 즉시 `require_escalated`와 짧은 승인 질문으로 재시도한다.
- 임시 staging 파일을 만들었다면 최종 복사 성공 후 Codex가 만든 임시 파일만 정리한다. 사용자가 만든 파일이나 기존 저장본은 명시 요청 없이 지우지 않는다.

## 패치 적용 원칙

- 큰 변경을 한 덩어리 패치로 넣지 않는다. 새 파일 추가, 지침 수정, 함수 내부 수정, 검증용 작은 보정은 가능한 한 별도 패치로 나눈다.
- `apply_patch`가 한 줄 실패로 전체 롤백될 수 있으므로, 넓은 범위를 고치기 전에는 `Select-String`/`Get-Content`로 정확한 현재 문맥을 확인한다.
- 함수 내부처럼 들여쓰기와 중괄호가 민감한 변경은 전체 함수 교체보다 조건문 1개, 파라미터 1개, 호출부 1개 단위로 작게 바꾼다.
- 단, 인코딩 깨짐 흔적이 남은 표시용 함수, 카운터 함수, 라벨 생성 함수처럼 범위가 작고 사용자 문구가 핵심인 함수는 깨진 문자열만 부분 보정하지 말고 함수 전체를 깨끗한 UTF-8 문구로 교체한다.
- 한글 힌트 문구, 안내문, 버튼 텍스트처럼 인코딩이나 문구 변경에 취약한 문자열은 패치 앵커로 우선 사용하지 않는다. 가능하면 `id`, 함수명, 클래스명, data 속성, 인접한 구조 코드처럼 안정적인 구조 앵커를 기준으로 수정한다.
- 한글 표시가 깨져 보여 전체 줄 매칭이 불안정하면 그 줄 전체를 old string으로 잡지 않는다. `id="..."`, `data-*`, `aria-*`, 함수명, enum 값처럼 ASCII 중심의 안정적인 속성값이나 식별자만 좁게 대상으로 삼아 교체한다.
- 함수 안에 한글 깨짐이 섞인 `title`, `label`, `placeholder`, `innerHTML` 문자열이 있으면 함수 전체나 큰 템플릿 블록을 한 번에 패치하지 않는다. 조건문, 배열 항목, 속성 할당, 함수 호출 같은 ASCII 중심 JS 구조 줄을 기준으로 작은 패치를 나눠 적용한다.
- 패치 실패 후에는 같은 큰 패치를 반복하지 말고, 성공 가능한 작은 부분부터 적용한 뒤 실패한 범위만 다시 읽고 재작성한다.
- 여러 파일을 동시에 바꿔야 해도 서로 의존하지 않는 파일은 분리 적용하고, 각 패치 뒤 최소한의 구문 검사나 검색 확인을 한다.

## 명령 quoting 회피

- `node -e`, PowerShell 한 줄 명령, cmd 인자처럼 따옴표와 백틱이 여러 겹 겹치는 인라인 명령은 짧고 단순할 때만 사용한다.
- 인라인 quoting이 한 번 실패했거나 한글 경로, 정규식, JSON, HTML, JS 코드 조각이 섞이면 같은 방식으로 반복하지 말고 워크스페이스 안에 짧은 임시 스크립트 파일을 만든 뒤 인자는 경로/접두어처럼 단순한 값만 넘긴다.
- SQL/Markdown 코드펜스처럼 백틱 문자가 들어가는 정규식을 PowerShell 인라인 명령에 직접 쓰지 않는다. 백틱이 필요하면 스크립트 안에서 `String.fromCharCode(96)` 또는 언어별 문자 코드 생성 방식으로 만들고, 정규식은 파일 안에서 조립한다.
- 검색 패턴에 따옴표, 백틱, 괄호, 역슬래시, 정규식 메타문자가 섞이면 PowerShell 인라인 quoting을 억지로 맞추지 않는다. 리터럴 검색이면 `Select-String -SimpleMatch`를 우선 쓰고, 복잡한 검색은 짧은 임시 스크립트나 패턴 파일로 분리한다.
- 임시 스크립트 파일명은 ASCII로 만들고, 작업 후 재사용 가치가 없으면 Codex가 만든 임시 파일만 정리한다.
- Playwright ad hoc 실행은 직접 `node -e`보다 `codex_playwright_node.cmd` 또는 `ensure_playwright_chromium.cmd` 래퍼를 우선 사용한다.

## 필수 데이터 규칙

- 각 레코드에는 `id`, `created_at`, `updated_at`을 둔다.
- 저장, 수정, 녹음 연결, 삭제, 학습상태 변경 등 모든 변경 시 `updated_at`을 갱신한다.
- Supabase에서 불러올 때는 `updated_at desc` 기준으로 가져온다.
- 같은 `id`가 로컬과 클라우드에 모두 있으면 `updated_at`이 최신인 쪽을 선택한다.
- 삭제까지 동기화하려면 `deleted_at` 또는 tombstone 테이블을 사용한다.
- 새 데이터 도메인이나 병렬 테이블을 추가하면 저장/로드, 화면 렌더링, 상세/편집, 삭제, tombstone, 일반 동기화, 최종본 저장, 백업/복원, 저장본 버전 확인을 함께 갱신한다.
- 새 데이터 도메인을 먼저 로컬 MVP로만 붙일 경우, 백업 메타나 개발자 정보에 `syncedToSupabase: false`처럼 클라우드 미연결 상태를 명확히 표시한다. SQL 테이블만 제공했거나 JSON 백업에 포함된 것만으로 Supabase 저장이 완성됐다고 보고하지 않는다.
- 새 데이터 도메인이 Supabase 저장 대상이라고 말하려면 최소한 원격 테이블/컬럼, row 정규화, 단건 upsert, batch upsert, 원격 load, tombstone/delete, 일반 동기화, 최종본 저장, canonical meta, 저장본 버전 확인, 전체 JSON 백업/복원이 모두 실제 코드 경로에 연결되어 있어야 한다.
- 앱 작동에 필수인 카탈로그, alias, 파서 기준값과 수시로 바뀌는 설명문, 해석문, 교육용 코멘트는 데이터 도메인을 분리한다. 후자를 Supabase 등 원격 테이블로 옮기면 JSON 백업/복원, 원격 pull/upsert/delete, 최종본 저장, manifest/hash, 저장본 버전 확인까지 함께 연결한다.
- 이미지·첨부파일 원본은 Postgres row나 브라우저 코드에 직접 넣지 않는다. 외부 스토리지를 쓰는 경우 Supabase에는 메타데이터만 저장하고, 브라우저에는 서비스 계정 키를 절대 넣지 않으며, 서버/Edge Function이 짧은 만료시간의 업로드·다운로드 URL을 발급하게 한다.
- 외부 이미지 저장소를 붙이면 업로드 성공만으로 완료하지 않는다. 편집 취소, 저장 실패, 기존 이미지 제거, 휴지통/보호기록 정리, 백업/내보내기까지 원본 파일과 메타데이터 수명주기가 함께 맞는지 확인한다.
- 내장 기본값을 원격 테이블로 이전할 때는 표시 기준과 마이그레이션 원본을 분리한다. Supabase를 기준 원본으로 정한 데이터는 화면 표시와 CSV/Excel 내보내기도 Supabase/사용자 저장값을 기준으로 하고, 내장값은 초기 업로드용 seed로만 남길지 명시한다.
- Supabase 전용 도메인이 아직 비어 있을 수 있으면 내보내기가 헤더만 있는 빈 파일이 되지 않게 한다. 저장값이 없을 때는 업로드용 seed/template CSV를 제공하거나, 빈 이유와 초기 업로드 버튼을 명확히 안내한다.
- 기능을 추가하거나 삭제할 때는 해당 버튼 하나만 보지 말고 관련 생명주기 전체를 점검한다. 예: 추가 기능이면 편집/삭제/일괄처리/백업/복원/동기화/검증 UI까지 확인한다.
- 내보내기/다운로드 버튼을 고친 경우 실제 선택 가능한 각 영역을 한 번씩 실행 경로로 검증한다. 특히 helper 함수명 변경 후 없는 함수 호출 때문에 버튼 전체가 멈추지 않는지 확인한다.
- 일괄 업로드용 양식 다운로드를 제공할 때는 사용자가 직접 맞추기 어려운 안정 식별자와 표시명을 미리 채운다. 예: `id`, `test_id`, `name`, `ko`, `sex`, `standard` 같은 키 필드는 비워두지 않는다.
- 참고치처럼 기관·검사실별로 추가되는 데이터 양식은 내장 기본 기준을 덮어쓰는 고정 양식으로 만들지 않는다. 양식 다운로드 전에 대상 기관명, 기준 id, 성별 범위 같은 기준 축을 명확히 선택하게 한다.
- 삭제 가능한 동기화 데이터는 활성 배열에서만 제거하지 말고 `deleted_at` 또는 tombstone을 남겨 다른 기기에서 되살아나지 않게 한다.
- 전체 JSON 백업/복원은 모든 주요 데이터 도메인과 tombstone을 포함해야 한다.
- 전체 JSON 백업/복원에 새 도메인을 추가하면 파일 객체에 포함되는지만 보지 말고 실제 UI 경로로 `생성 -> 백업 객체 확인 -> 현재 데이터 비움 -> 복원 적용 -> 목록/카드 재표시`까지 검증한다. 복원 후 메모리 배열에는 데이터가 있어도 프레임 DOM이 재생성되지 않아 화면이 비는 경우를 별도 실패로 본다.
- 사용자가 내려받는 백업/내보내기 파일명은 내부 개발명 대신 `백업날짜_앱이름_필드명` 형식을 우선한다. 파일명에 쓸 수 없는 문자(`/ \ : * ? " < > |`)는 안전하게 제거한다.
- 백업/복원/가져오기는 먼저 로컬 반영과 검증을 끝낸 뒤, 사용자가 일반 동기화 또는 최종본 지정을 선택하게 한다. 가져오기 도중 자동 flush/upsert가 섞이면 오래된 백업이 원격 최신본을 덮을 수 있다.
- 영역별 Excel/CSV 가져오기는 파일의 `updated_at`을 보존하고, 원격 최신 row가 더 최신이면 덮어쓰지 않는다. 가져오기 시각을 새 `updated_at`으로 임의 승격하지 않는다.
- 샘플 복원, 전체 삭제, 로컬 캐시 초기화는 명시적 목적이 아니면 `deviceId`, Supabase 설정, `syncMeta`, `canonical_version`을 바꾸지 않는다.
- batch upsert는 `fetch` 성공 여부만 보지 말고 반드시 `res.ok`를 확인한다.
- 연결 테스트 성공 시 성공 표시로 끝내지 말고 즉시 동기화를 실행한다.

## 일반 동기화

- 일반 자동 동기화는 로컬과 클라우드를 병합한 뒤 클라우드에도 upsert해서 로컬 전용 항목을 다른 기기로 전파한다.
- 단순 병합만 반복하면 기기별 항목 수가 계속 달라질 수 있다.
- 특히 한 기기에만 남은 오래된 로컬 전용 항목이 계속 되살아날 수 있으므로 최종본 기준 동기화와 섞지 않는다.
- JSON/CSV 대량 가져오기 중에는 outbox flush를 잠시 막고, 파싱/정규화/로컬 저장이 끝난 뒤 사용자가 선택한 모드에 따라 한 번만 동기화한다.
- 동기화 outbox/queue flush는 이미 실행 중일 때 즉시 성공을 반환하지 말고 기존 실행 Promise를 기다리게 한다.
- 원격 최신본 또는 원격 삭제와 충돌해 업로드를 건너뛸 때는 성공 처리로 outbox에서 지우지 말고 충돌 상태를 사용자에게 표시한다.
- 원격 설정이나 프로필 목록을 덮어쓴 뒤에는 현재 active id가 실제 항목을 가리키는지 반드시 보정한다.
- 원격 설정이나 프로필을 바꾸는 동안 편집 모달이 열려 있으면 모달 내부 화면도 새 active id 기준으로 다시 렌더링하거나 저장 대상을 고정한다. 화면은 A 프로필인데 저장 대상은 B 프로필이 되는 상태를 만들지 않는다.

## 로컬 캐시 저장소

- Supabase가 기준 원본인 앱에서 localStorage는 설정, device id, 작은 메타처럼 소량 데이터에만 사용한다.
- 용어, 약물, 문장, 학습상태, tombstone처럼 계속 커지는 로컬 캐시는 IndexedDB를 우선 사용한다.
- 앱 시작 시 로컬 캐시는 빠른 임시 표시용이며, Supabase 로드/동기화 성공 후의 화면을 기준본으로 본다.
- 로컬 캐시가 오래되었거나 저장 실패해도 Supabase 직접 저장이 성공했다면 데이터 원본은 보존된 것으로 본다. 다만 사용자가 혼동하지 않도록 로컬 캐시 실패/기준본 확인 중 상태를 안내한다.
- 기존 localStorage 대용량 캐시가 있으면 IndexedDB로 이관한 뒤 가능한 한 큰 localStorage 항목을 제거한다.
- IndexedDB 실패 fallback을 둘 때는 오래된 온전한 backup을 빈 데이터나 축소 데이터로 덮어쓰지 말고, 실패 상태 메타를 별도로 남긴다.

## 최종본 기준 동기화

- "이 기기 -> 클라우드" 같은 명령은 해당 기기를 최종본으로 지정하는 동작이다.
- 최종본 지정 시 Supabase 메타 테이블에 `canonical_version` 값을 갱신한다.
- 다른 기기는 `canonical_version` 변경을 감지하면 로컬 전용 항목을 보존하지 않고 클라우드 기준으로 맞춘다.
- `canonical_version`이 바뀐 경우에는 병합 결과를 다시 업로드하지 말고 클라우드 기준을 로컬에 반영만 한다.
- 클라우드 최종본 덮어쓰기/최종본 지정 버튼이 있는 앱은 근처에 read-only `저장본 버전 확인` 기능을 둔다.
- 저장본 버전 확인은 최종 저장 위치, 저장 시각, `canonical_version`/revision, 항목 수, 마지막 저장 기기, manifest/hash 검증 결과를 표시한다.
- 마지막 저장 기기는 UUID만 보여주지 말고 PC/태블릿/핸드폰 같은 기기 종류와 운영체제를 포함한 사람이 읽는 라벨을 병기한다.
- 기기 라벨은 추정값일 수 있으므로 안정 식별용 짧은 device id도 함께 표시한다.
- 사용자가 덮어쓰기 전에 현재 클라우드 기준본과 로컬 화면 캐시의 차이를 확인할 수 있어야 한다.
- 최종본 지정은 세션 같은 대표 도메인만 처리하면 안 된다. 사용자 편집 가능 도메인 전체에서 현재 기기에 없는 원격 전용 row를 delete 또는 tombstone 처리해 기준본을 완성한다.

권장 Supabase 메타 테이블:

```sql
create table if not exists language_sync_meta (
  key text primary key,
  value text,
  updated_at timestamptz default now()
);
```

## Supabase SQL 재확인

Supabase 동기화 관련 작업은 코드 수정 후 SQL 변경 필요 여부를 반드시 재확인한다.

확인 대상:

- 테이블
- 컬럼
- 타입
- primary key
- unique 제약
- RLS 정책
- Storage policy
- 메타 테이블과 `canonical_version` row

완료 보고에는 아래 중 하나를 반드시 명시한다.

```text
SQL 변경 필요: 있음
SQL 변경 필요: 없음
SQL 변경 필요: 불확실
```

Supabase 작업 후 SQL 확인 없이 완료 처리하지 않는다.
SQL 통합본이나 마이그레이션 SQL을 제공할 때는 파일 저장만으로 끝내지 말고, 사용자가 바로 복사해 Supabase SQL Editor에 붙여넣을 수 있도록 최종 답변에 전체 SQL을 `sql` 코드블록으로 함께 제공한다.
- 앱 안에 `SUPABASE_SCHEMA_VERSION`, `SUPABASE_SCHEMA_UPDATED_AT`, `SUPABASE_SCHEMA_SQL` 같은 스키마 안내 상수가 있으면, 테이블/컬럼/메타 row가 바뀌는 순간 앱 내부 상수, 개발자 정보의 “Supabase 동기화 스키마 설정 SQL”, 업데이트 이력, 최종 답변 SQL을 모두 같은 버전으로 맞춘다.
- 새 동기화 도메인을 추가했는데 앱 코드에는 REST 경로가 있고 통합 SQL에는 테이블이 빠지는 상태를 만들지 않는다. 예: `medical_notes` 같은 신규 테이블은 `create table`, `alter table add column`, primary key 보강, index, `language_sync_meta` canonical row, 진단 SELECT count/ready 항목까지 함께 넣는다.
- SQL을 JS template literal 안에 넣을 때 PostgreSQL dollar quote(`do $$ ... end $$;`)가 `$` 하나로 줄어들거나 `${...}` 보간으로 오염되지 않았는지 추출본을 다시 확인한다. SQL 제공 전에는 앱 내부 SQL에서 실제 복사될 텍스트를 추출해 `do $$`, `end $$;`, 신규 테이블명, 진단 SELECT가 그대로 남아 있는지 점검한다.
- SQL 버전명은 기능 업데이트 이력과 연결되는 이름으로 붙인다. 단순 `v1`, `최신`처럼 의미 없는 이름보다 `v4.13 · 단권화노트통합스키마`처럼 사용자가 예전/신규 스키마를 구분할 수 있는 이름을 우선한다.

## 대규모 변경 후 점검

- 구조, 저장, 동기화가 함께 바뀐 경우 완료 전 전수 점검을 한다.
- 점검 대상은 중복 안내, 모바일 표시, 상세 보기, 편집 모달 닫힘, 삭제 동기화, 백업/복원, 저장본 버전 확인, SQL 변경 필요 여부다.
- 백업/복원/최종본 저장을 바꾼 경우에는 전체 JSON, 영역별 CSV/Excel, tombstone, manifest/hash, 저장본 버전 확인, import 후 동기화 선택 UI를 함께 점검한다.
- 긴 입력 편집창은 배경 클릭으로 닫히지 않게 하고 명시적 닫기/저장/삭제 동작으로만 종료한다.
- 사용자 표시명을 바꾸면 버튼, 제목, 토스트, 빈 상태, 업데이트 이력까지 같은 이름으로 통일한다.

## UI 기능 추가 원칙

- 새 UI 기능은 먼저 기존 헬퍼 함수, 상태 변수, CSS 컴포넌트를 재사용할 수 있는지 확인한다.
- 웹앱의 색상, 배경, 카드, 테두리, 모달, 버튼은 하드코딩 색상보다 CSS 변수/토큰을 우선 사용해 다크모드와 라이트모드가 함께 깨지지 않게 한다.
- 라이트모드에서는 원색 버튼, 강한 채도의 텍스트, 이모지 고유색이 서로 충돌하면 조잡해 보일 수 있으므로 활성 버튼은 연한 tint 배경, 낮은 채도 텍스트, 얇은 테두리 중심으로 먼저 설계한다.
- 같은 기능군의 버튼은 화면 위치가 달라도 같은 색상 토큰을 공유한다. 예: 목록 상단 추가 버튼과 추가 모달의 저장/탭 버튼은 서로 다른 초록·파랑·보라 조합으로 흩어지지 않게 한다.
- 색상이 의미를 전달하지 않는 단순 액션 아이콘은 컬러 이모지보다 텍스트 기호나 기존 아이콘 컴포넌트를 우선 검토한다. 이모지를 쓰면 다크/라이트 양쪽에서 주변 텍스트색과 충돌하지 않는지 확인한다.
- 다크/라이트 전환을 추가하거나 수정하면 본문 카드뿐 아니라 모달, 검색 결과, 입력창, 고정 하단 dock, 그래프/SVG, 상태 배지, 비활성 버튼까지 두 모드에서 잔여 고정색이 없는지 확인한다.
- H/L/정상 수, 경고 수, 오류 수처럼 요약 배지에 숫자를 표시하면 가능한 한 클릭 시 해당 항목명, 값, 날짜를 확인하는 상세창을 제공하고, 상세 항목에서 원래 row/카드/추이 화면으로 이동하게 한다.
- 상세 드로어, 상세 카드, 결과 패널처럼 정보량이 많은 화면은 label/value를 단순 세로 나열하지 않는다. 상단 요약, 핵심 태그, 기본 정보, 임상/학습 포인트처럼 정보 층위를 나누고, 반복 필드는 카드/그리드/섹션 제목으로 줄간격과 시선 흐름을 정리한다.
- 장문 편집기나 단권화 노트에 표 삽입 기능을 넣을 때는 표가 단순 격자처럼 보이지 않게 머리행 음영, 얇은 테두리, 적절한 셀 패딩, 줄무늬 또는 행 간격으로 읽는 방향을 만든다. 다크/라이트 모드 모두에서 머리행·본문·테두리 대비가 과하거나 탁하지 않은지 확인한다.
- 장문 편집기나 단권화 노트 본문에 이미지를 직접 삽입하면 편집기 삽입 버튼뿐 아니라 sanitizer 허용 태그/속성, 저장 메타데이터, 상세 보기 중복 갤러리, 첨부 삭제 시 본문 참조 제거까지 함께 점검한다.
- 표 편집 UX는 삽입 후 첫 빈 셀에 커서가 들어가고, 표 뒤에 빈 단락을 자동으로 만들어 사용자가 표 다음에 바로 글을 이어 쓸 수 있게 한다. Tab 이동, 마지막 셀에서 행 추가, 행/열 삽입·삭제 같은 워드식 기대 동작을 넣는 경우 마지막 1행/1열 삭제 방지 같은 안전장치도 함께 둔다.
- 표 행/열 조작 메뉴, 리사이즈 핸들, 선택 표시 같은 편집 전용 UI는 저장되는 본문 HTML에 남기지 않는다. 저장 전 sanitize/cleanup 경로에서 편집 위젯 속성·클래스·임시 버튼을 제거하고, 자동 링크·백업·동기화가 순수 본문 표만 다루게 한다.
- 모바일·태블릿에서 표가 들어간 카드나 편집기는 `overflow-x:auto` 또는 반응형 축소 규칙을 두고, `document.scrollingElement.scrollWidth <= innerWidth` 검증으로 전체 페이지 가로 스크롤이 생기지 않는지 확인한다. 표 내부 스크롤은 허용하더라도 페이지 전체가 밀리면 실패로 본다.
- 새 병렬 도메인을 추가하면 목록 카드뿐 아니라 오른쪽 상세보기, 모바일 bottom sheet, 편집 모달의 정보 위계와 글자 크기를 기존 주요 도메인과 같은 품질 기준으로 맞춘다. 새 도메인만 임시 목록형 UI로 남기지 않는다.
- 반응형 CSS에서 전역 버튼 규칙을 추가할 때는 상단 toolbar, 하단 dock, 모달 버튼처럼 같은 `.btn`/`.qbtn` 계열을 공유하는 영역이 함께 영향을 받는지 확인한다. 태블릿 breakpoint와 모바일 breakpoint는 별도로 검증하고, 태블릿에 모바일 3분할 하단바가 과하게 적용되지 않게 한다.
- 모바일·태블릿 모달 안에 접힘형 입력기, 수식 입력기, 특수문자 패널을 넣으면 펼친 직후 패널이 현재 viewport 안에 들어오는지 확인한다. 패널이 아래로 걸치면 사용자가 한 번 더 스크롤해야 하므로 `scrollIntoView({ block: 'nearest' })` 같은 보정이나 더 가까운 배치를 적용한다.
- 입력 중인 검색창이나 필터 UI를 `innerHTML`로 다시 그리는 경우 포커스와 커서 위치가 사라지는지 확인한다. 가능하면 결과 영역만 다시 렌더링하고, 불가피하면 `requestAnimationFrame`으로 focus/selection을 복원한다.
- 검색 결과, 자동완성, 자동링크처럼 클릭 시 모달 위에 drawer/bottom sheet를 여는 UI는 데스크톱 `click`만으로 검증하지 않는다. 모바일 터치에서는 `pointerdown`에서 새 overlay를 열면 손을 떼며 발생하는 `click`이 overlay에 떨어져 즉시 닫힐 수 있으므로, 열기 동작은 가능하면 `click` 한 경로로 두고 `mousedown.preventDefault()`는 blur 방지 용도로만 쓴다.
- 검색 입력칸에서 상세 drawer/bottom sheet를 열 때는 열기 직전에 입력칸 `blur()`를 고려한다. 모바일 키보드가 열린 채로 있으면 bottom sheet 위치와 viewport 계산이 흔들릴 수 있다.
- 클릭 가능한 카드 안에 버튼을 넣을 때는 이벤트 버블링으로 카드 열기/닫기 동작이 섞이지 않도록 `stopPropagation`을 확인한다.
- 목록 카드, 검색 결과, 자동링크에서 상세 대상을 열 때 id는 숫자라고 가정하지 않는다. localStorage/JSON/Supabase/import 경로를 거치며 문자열 id가 섞일 수 있으므로 비교는 `String(a) === String(b)`처럼 안정화하거나 한 곳에서 정규화한다.
- 브라우저 내장 기능을 쓰는 경우 지원 여부를 감지하고, 미지원·빈 입력·권한 실패를 토스트나 화면 안내로 표시한다.
- AI, 외부 API, 서버 동기화처럼 키나 설정이 필요한 기능은 실제 유효한 키/설정이 있을 때만 `활성화됨`으로 표시한다. 키가 없으면 네트워크 호출이나 재시도를 시작하지 말고 즉시 기본 저장, 수동 입력, 안내 토스트 같은 비AI fallback으로 빠져야 한다.
- 날짜나 레코드를 저장·수정한 직후에는 목록/대시보드가 방금 저장한 항목을 선택 상태로 유지하게 한다.
- 사용자가 입력하거나 기본값으로 채우는 날짜는 `toISOString().slice(0,10)`로 만들지 않는다. UTC 기준이라 한국 시간 등 로컬 자정 전후에 전날로 밀릴 수 있으므로 `getFullYear/getMonth/getDate` 기반 로컬 날짜 helper를 사용한다.
- 외부 스크립트를 동적으로 로드할 때는 이미 태그가 있다는 이유만으로 성공 처리하지 않는다. 로딩 중복 호출을 막기 위해 `src -> Promise` 캐시를 두고, 로드 완료와 로드 중 상태를 구분한다.
- 데이터 구조를 바꾸지 않는 UI-only 변경도 완료 전 Supabase SQL 변경 여부를 확인하고 보고한다.
- 프론트엔드 변경 후에는 JS 구문 검사와 가능하면 실제 브라우저 렌더링 확인을 수행한다. 레이아웃 변경이면 최소 데스크톱, 태블릿, 모바일 폭을 나눠 확인한다. 브라우저 검증이 환경 문제로 실패하면 그 사실을 완료 보고에 남긴다.
- 슬라이드 서랍, bottom sheet, 접힘 패널처럼 transition이 있는 UI는 class 적용 직후의 중간 위치만 보지 말고 애니메이션 완료 후 최종 rect도 확인한다.
- 모달 안에서 상세 drawer/bottom sheet를 다시 여는 UI는 `z-index` 값을 눈대중으로 정하지 않는다. 열린 뒤 `getComputedStyle(...).zIndex`와 `getBoundingClientRect()`로 대상 패널이 모달 overlay보다 앞이고 viewport 안에 있는지 데스크톱·태블릿·모바일 각각 확인한다.
- 로컬 캐시나 Supabase가 없는 headless 브라우저에서 카드 수가 0개인 상태로 렌더링 테스트가 끝나면 상세보기·편집·삭제·입력기 검증을 통과한 것으로 보지 않는다. 필요한 경우 테스트 안에서 최소 샘플 데이터를 주입한 뒤 실제 UI 함수와 클릭 경로로 다시 검증한다.
- 반응형 레이아웃 검증은 `document.scrollingElement.scrollWidth <= innerWidth` 같은 수평 스크롤 여부, 주요 패널의 `getBoundingClientRect()`가 viewport 안에 있는지, 닫기/저장 같은 핵심 버튼이 보이는지를 함께 본다.
- Playwright 검증이 필요할 때는 사용자가 수동으로 루트 이동, 패키지 설치, Chromium 설치를 지시하게 두지 않는다. Codex가 먼저 `package.json`/`playwright.config.*` 기준 실제 프로젝트 루트와 lockfile 기준 패키지 매니저를 찾고, Playwright 패키지와 Chromium 실행 파일을 확인한 뒤 필요한 설치를 승인 요청해 실행한다.
- 단일 HTML 앱처럼 `package.json`이 없으면 작업 폴더의 `ensure_playwright_chromium.ps1`로 Codex 번들 Playwright를 사용해 Chromium을 보장하고, 필요 시 최신/지정 HTML을 headless smoke test로 검증한다.
- 브라우저 검증에서 `file://` 직접 열기가 보안 정책이나 실제 배포 환경 차이로 막히면 우회하려 하지 않는다. 워크스페이스 폴더를 127.0.0.1 로컬 서버로 띄우고 `localhost`/loopback URL로 검증한다.
- 브라우저 검증 스크립트는 앱의 최상위 `const`/`let`이 `window.X`로 노출된다고 가정하지 않는다. 대기 조건은 가능한 한 DOM 표시, 접근 가능한 앱 함수, localStorage/IndexedDB 상태, 명시적 test hook처럼 실제 구조에 맞는 신호로 잡고, 내부 전역명 하나에 과하게 의존하지 않는다.
- `TERMS`, `MICROBES`처럼 앱 내부 `let`/`const` 데이터 접근은 readiness 대기 조건으로 쓰지 않는다. 화면 준비 대기는 DOM/공개 함수 기준으로 끝내고, 내부 데이터 검증은 그 뒤 별도 `evaluate` 단계에서 접근 가능한 경로를 확인하거나 공개 test hook을 추가한다.
- 단일 HTML에서 IIFE나 클로저 안에 숨겨진 함수는 테스트에서 직접 호출할 수 있다고 가정하지 않는다. 복원/저장/렌더링 검증은 공개 UI 함수, 버튼 클릭, 파일 import 흐름, 탭 전환 같은 실제 사용자 경로로 수행하고, 필요하면 최소 test hook을 명시적으로 노출한다.
- 탭 전환이나 `mainContent.innerHTML` 교체가 있는 앱은 복원 후 데이터 배열 개수만 확인하지 않는다. 대상 프레임 DOM이 사라졌을 때도 `ensure*Shell` 같은 재생성 경로가 작동하는지, 실제 카드 텍스트가 다시 렌더링되는지 확인한다.
- 브라우저 탐색이 시간 초과되면 즉시 앱 렌더링 오류로 단정하지 않는다. 먼저 로컬 서버가 해당 URL에 응답하는지 짧은 HTTP 요청으로 확인하고, 서버는 정상인데 탐색만 멈추면 새 page/context 또는 새 포트로 한 번만 짧게 재시도한 뒤 결과를 분리해 보고한다.
- 비승인 실행에서 Chromium 확인/설치가 시간 초과되면 같은 명령을 반복하지 않는다. `ensure_playwright_chromium.cmd -CheckOnly`처럼 빠른 로컬 경로 확인만 먼저 수행하고, 설치·브라우저 실행·AppData 캐시 접근이 필요하면 즉시 승인 실행으로 전환한다.
- Chromium 바이너리가 실제로 없어서 최초 설치가 필요하면 2분 안에 끝난다고 가정하지 않는다. 다운로드형 설치는 처음부터 승인 실행과 긴 제한 시간(최소 5분, 느린 네트워크면 10분)을 사용하고, 시간 초과 후에는 즉시 반복하지 말고 `-CheckOnly`로 설치 완료 여부를 먼저 확인한 뒤 한 번만 긴 제한으로 마무리 실행한다.
- Codex 번들 Node로 Playwright를 직접 `node -e` 실행할 때는 `NODE_PATH` 누락으로 `playwright-core` 해석이 빗나가지 않게 `codex_playwright_node.cmd`를 우선 사용한다. 직접 실행해야 한다면 번들 `node_modules`, `node_modules\.pnpm\node_modules`, `node_modules\playwright\node_modules`, `.pnpm\playwright@*\node_modules`, `.pnpm\playwright-core@*\node_modules` 후보를 함께 넣는다.
- 한글·공백이 포함된 HTML 파일 경로를 PowerShell 파이프, `node -e`, cmd 인자에 직접 길게 넘기지 않는다. 가능하면 `ensure_playwright_chromium.cmd -RunTests -HtmlPrefix 260622_09.19`처럼 ASCII 접두어를 넘기고, 스크립트 내부에서 최신 HTML을 찾게 한다.
- Playwright 완료 보고에는 실제 프로젝트 루트, 패키지 매니저, Playwright 패키지 여부, Chromium 바이너리 여부, 실행한 설치 명령, 실행한 테스트 명령, 남은 오류를 포함한다.

## 수치·단위 데이터 검증 원칙

- 검사값, 점수, 금액처럼 단위나 기준범위가 붙는 데이터는 표시용 반올림값을 저장하지 말고 원 수치를 보존한다.
- 외부 JSON, 과거 캐시, 원격 row처럼 앱 밖에서 들어오는 레코드는 렌더링 전에 정규화한다. 숫자 값 map, 정성 판정 map 같은 중첩 객체가 `null`, 배열, 문자열이면 빈 객체로 보정해 렌더링 크래시를 막는다.
- 자동판독, OCR, AI 추출 결과는 숫자만 받지 말고 가능하면 원 단위를 함께 받아 앱 기준 단위 변환 함수를 반드시 통과시킨다.
- 단위 문자열은 범용 단위보다 구체 단위를 먼저 판정한다. 예: `μIU/mL`을 `IU/mL`보다 먼저, `mmol/mol`을 `mmol/L`보다 먼저 처리한다.
- PDF, 이미지, OCR처럼 추출층이 불안정한 입력은 "잡힌 항목 수"만 보지 말고 핵심 필수 항목의 누락 여부를 별도로 검증한다.
- 핵심 항목이 누락될 수 있는 문서 자동판독은 1차 텍스트 추출, 제한적 OCR fallback, 사용자 확인/수정 입력의 단계로 설계한다. AI fallback은 마지막 선택지로 둔다.
- 새 결과지 양식이나 외부 샘플을 반영하면 실제 샘플 기반 회귀 테스트를 남기고, 기대값에는 단위 변환 후 앱 기준값까지 포함한다.
- 참고치, 설정, 사용자 편집값처럼 일부 필드만 수정 가능한 데이터는 누락된 필드를 `null`로 확정하지 말고 기본값 fallback 여부를 검토한다.
- 사용자가 직접 입력한 참고치 low/high와 앱 내 경계판정 threshold가 모두 있는 항목은 어느 값이 H/L 판정에 우선하는지 명시하고 테스트한다.
- 저장본 버전 확인용 manifest/hash는 모든 사용자 편집 가능 도메인과 참고치 설정을 포함해야 한다.
- 외부 버그 리포트나 사용자 제보를 반영할 때는 주장별로 실제 코드 경로를 대조하고, 재현 예문을 함수 단위 테스트로 만든 뒤 수정한다.
- 참고범위만 적힌 `0-3 /HPF`, `0-40` 같은 range-only 숫자를 실제 결과값으로 자동 저장하지 않게 파서 테스트에 포함한다.
- 한 번이라도 실제로 발생한 오작동은 단순 수정으로 끝내지 말고 재발 방지 지침과 회귀 테스트에 남긴다.
- 과거 실수와 같은 범주의 변경을 할 때는 기존 실패 예문이 계속 통과하는지 확인하기 전에는 완료 처리하지 않는다.
- "잡힌 항목 수"가 늘어나는 것만 성공으로 보지 말고, 과거에 틀렸던 핵심 항목의 실제 값이 맞는지 별도로 대조한다.
- 참고치, 단위, 날짜, 결과값처럼 서로 붙어 나오는 문서는 `값 앞 참고치`, `값 뒤 참고치`, `참고치만 있음`, `결과값만 있음`을 모두 테스트한다.
- OCR/PDF 숫자 전처리는 소수점 조각을 복원하더라도 참고범위와 결과값 사이의 경계를 없애지 않는다. 예: `3.4~7 4.9`를 `3.4~74.9`처럼 붙이면 안 되며, 전처리 변경 후에는 원문 토큰과 파싱 결과를 함께 검증한다.
- 병원별/양식별 템플릿 파서가 있는 문서는 전용 파서를 일반 alias scan보다 우선하고, 전용 파서가 해당 항목명을 감지했지만 결과값을 신뢰하지 못하면 다른 줄의 참고치 숫자로 fallback하지 않는다.
- PDF 검증은 개발용 추출기 하나만 믿지 않는다. 실제 앱이 브라우저 `pdf.js`로 행을 재구성한다면 같은 경로로 `parseLabText()`까지 돌려 확인하고, PyMuPDF/pdfplumber 등 다른 추출기 통과만으로 완료하지 않는다.
- `pdf.js`가 글자 단위로 분절한 텍스트는 한글-한글뿐 아니라 라틴-한글 경계 공백도 별도로 검토한다. 예: `B 형간염 S 항원`, `C 형간염 항체`, `A 형간염 항체`처럼 라틴 문자와 한글 사이가 벌어진 라벨은 전용 space-tolerant 패턴과 실제 PDF 회귀 테스트를 둔다.
- Positive/Negative 같은 정성검사 파서는 현재 결과를 찾기 전에 `미만`, `이하`, `reference`, `참고치`에 붙은 숫자와 판정어를 먼저 걸러야 한다. 병원별 섹션 파서가 해당 정성검사 항목을 신뢰 가능하게 찾았으면 일반 alias scan보다 먼저 authoritative하게 반영한다.
- 표 형태 문서에서 검사항목명, 단위, 참고범위, 결과값이 여러 줄로 쪼개질 수 있으므로 다음 검사항목 전까지만 합친다. 다만 후보 선택은 짧은 후보부터 검사하고, 참고범위만 있는 후보처럼 결과값이 부족할 때만 다음 줄을 덧붙인다.
- 한글 검사항목 정규식에는 영어식 단어 경계 `\b`를 무심코 붙이지 않는다. `총단백\b`, `음성\b`처럼 한글 뒤 경계가 실패할 수 있으므로 공백, 괄호, 문자열 끝 같은 실제 구분자를 명시한다.
- 줄바꿈으로 떨어진 고아값을 역방향/주변행 fallback으로 붙일 때는 표 머리글, 날짜행, 참고기준치 행, 설명문을 먼저 배제한 뒤 값 후보를 검토한다.
- 약어가 겹치는 항목은 짧은 alias보다 구체 항목명을 먼저 매칭하고, 충돌 가능성이 있는 alias는 앱별 스킬 파일에 명시적 테스트로 둔다.
- 같은 줄에서 구체 항목으로 이미 판정된 값이 더 넓은 alias에 다시 걸리지 않게 한다. 예: `당화혈색소` 줄은 HbA1c로 처리하고 Hemoglobin alias fallback이 같은 숫자를 다시 저장하지 않게 한다.
- 파서 수정 후에는 전체 인식 개수뿐 아니라 과거 오인식된 핵심 항목의 기대값을 앱별 스킬 파일의 회귀 예문으로 직접 대조한다.

## 핵심 교훈

- 로컬 전용 항목 보존은 데이터 유실 방지에는 좋지만 최종본 기준 동기화에서는 위험하다.
- 마지막 작업기기 기준 동기화와 양방향 병합 동기화는 서로 다른 모드로 설계해야 한다.
- 병렬 데이터 도메인은 화면에 보이는 저장만으로 끝나지 않고 삭제, 백업, 복원, 최종본 검증까지 같은 수준으로 다룬다.
- 백업 파일을 가져오는 행위와 클라우드 기준본을 바꾸는 행위는 분리한다. 가져오기는 로컬 변경이고, 원격 반영은 사용자가 선택한 동기화 모드에서만 실행한다.
- 최종본 저장은 "있는 것 upsert"만으로 부족하다. 현재 기준본에 없는 원격 전용 데이터까지 정리해야 다른 기기에서 과거 데이터가 되살아나지 않는다.
